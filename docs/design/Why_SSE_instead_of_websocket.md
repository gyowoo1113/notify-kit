## Why SSE instead of WebSocket

notify-kit의 주요 사용 사례는 **알림 전달**이며, 이는 대부분 서버에서 클라이언트로의 단방향 전송
이 요구사항에 가장 적합한 방식으로 SSE를 선택하였음

### 선택 기준

- 알림은 단방향 통신이므로 WebSocket의 양방향 특성이 필수적이지 않음
- SSE는 HTTP 기반으로 Spring MVC 환경에서 간단히 적용 가능함
- 브라우저에서 `EventSource`를 통해 별도 설정 없이 사용 가능함
- 라이브러리 형태로 제공할 때 통합 및 운영 부담이 상대적으로 적음

### 고려 사항

- SSE는 기본적으로 단방향 통신이므로 클라이언트 → 서버 메시지가 필요한 경우 부적합함
- 다중 인스턴스 환경에서는 추가적인 확장(예: Redis Pub/Sub)이 필요할 수 있음

> 본 프로젝트는 SSE를 최소 실시간 전송(MVP)으로 제공하며,
>  필요 시 Outbox 패턴 및 메시징 시스템으로 확장 가능한 구조를 유지함
